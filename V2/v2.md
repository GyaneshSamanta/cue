## 1. What v1.0 Got Right (and Where Its Ceiling Is)

Before designing v2.0, it is worth being honest about what the first version establishes, and where its natural limits are. This prevents the common mistake of adding features that paper over architectural gaps rather than solving real friction.

### 1.1 v1.0 Strengths

- **Zero-dependency local execution.** The core promise holds — no LLM runtime, no cloud call during normal use, single binary.
- **Lock queuing solves a real and recurring pain.** On Linux especially, `dpkg` lock conflicts are one of the most common beginner stumbling blocks. This feature has immediate, visible value.
- **Pause/resume is differentiating.** No existing CLI tool does this for package installations. It's the feature that makes people share the tool.
- **Environment Stores reduce a 2-hour manual process to one command.** The composability of 7 stacks covers the majority of developer personas.
- **History with project tagging is underrated.** Most developers don't realize how useful a queryable, tagged history is until they lose it.

### 1.2 v1.0 Gaps That v2.0 Must Address

| Gap | User Experience Today | What v2.0 Should Do |
|-----|----------------------|---------------------|
| **First-run experience is cold** | User installs the binary; nothing happens; they read docs. | Interactive onboarding wizard. |
| **No à la carte tool installs** | Stores install full stacks; you can't just install "Node.js via nvm" alone. | Toolkit installer — single tool, right way, right version manager. |
| **Claude Code setup is genuinely painful** | Requires knowing about MCP, Node.js, API keys, model config — too many moving parts. | Dedicated guided store with and without Ollama. |
| **CLI is purely text; no interactivity** | Every selection is done by memorizing sub-command names. | TUI: arrow keys, searchable menus, live previews. |
| **No project-type detection** | User must know which store applies to their project. | Auto-detect from files on disk. |
| **Macros are static; no extension mechanism** | Community cannot contribute macros without forking. | Plugin system with versioned TOML packs. |
| **No version switching** | Stores install one version; no way to switch `node@18` to `node@20`. | Version manager integration. |
| **Only 7 stacks** | Mobile, DevOps, Rust, Game Dev, Web3, Java users are unserved. | Seven new stacks. |
| **Workspace backup is one-size-fits-all** | No per-machine config variance; no team sharing. | Profiles + team sync. |

## 2. v2.0 Vision Statement

> v1.0 made the terminal less painful. v2.0 makes it feel intelligent — not because it uses AI, but because it understands the shape of developer work well enough to anticipate, guide, and remember.

The two design principles that govern every v2.0 decision:

1. **The right default is better than a configurable option.** When a user types `cue store install mern`, the right thing should happen without flags. Configuration is for the 20% who need to override it.
2. **Every command should be discoverable by a first-time user without reading documentation.** If a user has to open a README to use a feature, the feature is not finished.

## 3. Pillar 1 — UX Overhaul

This is the highest-leverage pillar in v2.0. The underlying engine from v1.0 is sound; the surface layer is what needs the most work. A user who understands the tool's capabilities from the first minute retains and evangelizes; a user who has to discover features through documentation doesn't.

### 3.1 First-Run Wizard (`cue setup`)

On first launch (detected via absence of `~/.cue/config.toml`), `cue` automatically enters setup mode. Also triggerable explicitly via `cue setup --reset`.

**Wizard Flow:**

```
Welcome to cue v2.0

Let's configure a few things. This takes about 60 seconds.

[1/5] Detected: Ubuntu 24.04 (apt) — is this right? [Y/n]

[2/5] What shell do you use? (detected: zsh)
  ● zsh  ○ bash  ○ fish  ○ PowerShell

[3/5] Would you like shell completions installed? [Y/n]
  → Will add source ~/.cue/completions.zsh to your .zshrc

[4/5] Enable desktop notifications when queued commands finish? [Y/n]

[5/5] Would you like to set a default project tag for this machine?
  (e.g. "work", "personal", "ml-projects")  Leave blank to skip: _

Setup complete. Run 'cue doctor' to verify your environment.
```

The wizard writes `config.toml`, installs completions, and prints a summary of 5 useful commands to try next.

### 3.4 `cue diff <stack>` — Environment Gap Analysis

Shows the delta between what a store expects and what is currently installed. Critical for onboarding to an existing project on a new machine.

```
cue diff mern

Comparing current environment vs MERN store requirements...

  PRESENT   ✔  node        v20.11.0  (required: LTS)
  PRESENT   ✔  npm         10.2.4    (required: ≥8.0)

  MISSING   ✗  newman      —         not installed

```

The `--missing-only` flag installs only what is absent; `--upgrade` handles outdated items.

### 3.5 Shell Hook Integration (`cue shell-hook`)

Adds smart hooks to the user's shell profile that activate contextual behaviour when navigating directories.

**What hooks do:**

- **Auto-activate virtual environments.** When `cd` into a directory with `.venv/` or `venv/`, automatically activate it. On `cd` out, deactivate.
- **Auto-switch project tag.** When `cd` into a directory with a `.cue` config file, set the active history tag automatically.
- **Prompt enrichment.** Optionally show the active project tag in the shell prompt.
- **Stack detection on entry.** If the directory has a `package.json` / `requirements.txt` / `Cargo.toml` / `go.mod`, print a one-line hint about missing tools.

**Installation:**

```
cue shell-hook install
  → Adds 3 functions to ~/.zshrc (or ~/.bashrc / config.fish)
  → Requires shell restart or: source ~/.zshrc

cue shell-hook uninstall
  → Removes all hooks cleanly
```

### 3.7 Improved Error Messages — The "Fix It" Pattern

Every error in v2.0 follows this structure:

```
✖ [ERROR] Cannot install mongodb: package manager is locked.

  What happened:
  dpkg is holding a lock at /var/lib/dpkg/lock-frontend.
  This usually means an automatic system update is running.

  What to do:
  1. Let cue queue this (recommended):
     Your command will run automatically when the lock clears.
     Press Enter to queue, or Ctrl+C to cancel.

  2. Check what's holding the lock:
     sudo lsof /var/lib/dpkg/lock-frontend

  3. Force-clear (if nothing important is running):
     sudo rm /var/lib/dpkg/lock-frontend  ← use with caution
```

The pattern: **What happened** (specific, not generic) then **What to do** (numbered, concrete, ranked by safety).

## 4. Pillar 2 — Toolkit Installer (New Concept)

### 4.1 The Distinction from Stores

**Stores** are the combo meal — install an entire stack in one command. **Toolkits** are à la carte — install a single tool, the right way, using the right version manager, without the surrounding ecosystem.

The canonical problem toolkit solves:

> "I just need Node.js. I don't want the whole MERN store. But I also don't want to Google 'how to install Node.js with nvm on macOS' for the fifth time."

`cue toolkit install node` does exactly this — installs Node.js via `nvm`, configures it correctly, adds the right lines to shell config — and nothing else.

### 4.2 Command Surface

```bash
cue toolkit install <tool>          # Install latest stable
cue toolkit install <tool>@<ver>   # Install specific version
cue toolkit upgrade <tool>          # Upgrade to latest
cue toolkit remove <tool>           # Uninstall
cue toolkit list                    # All available tools
cue toolkit list --installed        # Only installed tools
cue toolkit info <tool>             # Version, install method, size
```

### 4.4 Version Manager Bootstrap — The Right Way

Each language toolkit bootstraps its version manager first, then uses it to install the language. Installing Python directly via `apt` instead of `pyenv` means version conflicts within months.

```
cue toolkit install python

  [1/3] Installing pyenv (if not present)
        → Cloning pyenv to ~/.pyenv
        → Adding pyenv init to ~/.zshrc
  [2/3] Installing Python 3.12.2 (latest stable)
        → pyenv install 3.12.2
        → pyenv global 3.12.2
  [3/3] Installing pip + pipx

✔ Python 3.12.2 installed.
  Switch versions later: cue version use python 3.11.8
```

## 5. Pillar 3 — Claude Code Store

This is the most complex store in v2.0 and deserves its own pillar. Claude Code setup has a reputation for being painful — multiple moving parts, version conflicts, unclear MCP configuration, and the added complexity of choosing between API-backed and fully local (Ollama) modes.

`cue store install claude-code` eliminates all of that friction with a guided, hardware-aware, mode-aware installation.

### 5.1 Two Installation Modes

```
cue store install claude-code

How would you like to use Claude Code?

  ● API mode   — Full Claude capability via Anthropic API
                 Requires: ANTHROPIC_API_KEY
                 Cost: Pay per token (standard Anthropic pricing)

  ○ Local mode — Fully offline via Ollama local models
                 Requires: ~4–8 GB RAM, no API key
                 Cost: Free (runs on your machine)

  ○ Hybrid     — API online, Ollama as offline fallback
                 Best of both worlds
```

### 5.3 Claude Code — Local Mode (with Ollama)

The local mode targets developers who want a fully offline AI coding assistant. The core technical challenge is that Claude Code speaks the Anthropic API format, while Ollama exposes an OpenAI-compatible format. The bridge is **LiteLLM Proxy** — a lightweight Python server that translates between the two.

`cue` installs and configures this entire chain automatically.

**Installation Flow:**

```
Step 1/8  Prerequisites — Node.js + Python checked/installed

Step 2/8  Ollama
  → Installing Ollama
  ✔ Starting Ollama service (systemd / launchd / Windows service)

Step 3/8  Model Selection (hardware-aware)
  Your hardware: 16GB RAM, RTX 3050 Ti (4GB VRAM)
  ⚠ 4GB VRAM — models will run in CPU+RAM hybrid mode (~8-12 tok/s)
  Tip: phi3.5-mini:3.8b fits fully in VRAM for faster inference

  ✔ qwen2.5-coder:7b  (recommended for code — 4.7 GB)
  ? deepseek-r1:8b    (strong reasoning — 5.2 GB) [y/N]

  Pulling qwen2.5-coder:7b...
  [████████████████░░░░] 74% — 3.5 GB / 4.7 GB — 2m 30s remaining

Step 4/8  LiteLLM Proxy (Ollama → Anthropic API bridge)
  → pip install litellm[proxy]
  ✔ Config written to ~/.cue/litellm-config.yaml
  ✔ LiteLLM installed as a user service (auto-starts on login)
  ✔ Proxy running at http://localhost:4000

Step 5/8  Claude Code CLI
  → npm install -g @anthropic-ai/claude-code

Step 6/8  Routing Claude Code to local proxy
  ✔ ANTHROPIC_BASE_URL=http://localhost:4000 → ~/.zshrc
  ✔ ANTHROPIC_API_KEY=local-mode → ~/.zshrc (required placeholder)
  ✔ Default model: qwen2.5-coder:7b → ~/.claude.json

Step 7/8  MCP Servers (optional — same as API mode)

Step 8/8  Open WebUI (optional graphical chat interface)
  ? Install Open WebUI? [y/N]: y
  → pip install open-webui
  ✔ WebUI available at http://localhost:3000

Verification
  ✔ Ollama running, models loaded
  ✔ LiteLLM proxy responding
  ✔ Claude Code routing to local model
  ✔ Test completion successful

Setup complete.
  Code assistant : claude "refactor this to use async/await"
  Chat UI        : open http://localhost:3000
  Add models     : cue model pull <name>
```

### 5.4 Hybrid Mode

Installs both API and local setups. LiteLLM proxy handles automatic switching:
- **Online:** Route to Anthropic API (full Sonnet/Opus capability).
- **Offline (detected):** Fall back to local Ollama model automatically.

The `fallback_models` key in the LiteLLM config is written automatically by `cue`.

## 7. Pillar 5 — New Core Feature Modules

### 7.1 Plugin System (`cue plugin`)

Allows the community to distribute macro packs and custom stores without forking.

**Plugin Format** — a TOML file hosted anywhere:

```toml
[meta]
name = "docker-advanced"
version = "1.2.0"
description = "Advanced Docker macros for production workflows"
author = "community"
min_cue_version = "2.0.0"

[[macro]]
name = "docker-compose-clean"
command = "docker compose down --volumes --remove-orphans && docker compose build --no-cache && docker compose up -d"
dangerous = true
explanation = """
Performs a full clean restart of your compose stack.
ALL volumes are deleted (database data included).
Images are rebuilt from scratch ignoring cache.
Use this when your stack is in an unknown state.
"""
```

**Plugin Commands:**

```bash
cue plugin install docker-advanced        # From curated registry
cue plugin install https://gist.github.com/user/abc  # From URL
cue plugin list
cue plugin remove <name>
cue plugin update
cue plugin create     # Interactive scaffolding wizard
cue plugin publish    # Submit to curated registry
```

**Security Model:**
- Plugins are TOML-only — no binary downloads, no arbitrary script execution outside macro command strings.
- First run of any plugin macro shows the exact commands that will execute and requires confirmation.
- Plugins cannot register daemons or startup items.
- All plugin macro executions are tagged `source = "plugin:<name>"` in `history.db`.

---

### 7.2 Version Manager Integration (`cue version`)

A unified interface for switching between installed versions of any runtime, regardless of which version manager backs it.

```bash
cue version list python           # pyenv-installed Python versions
cue version list node             # nvm-installed Node versions
cue version use python 3.11.8    # Switch global Python
cue version use node lts         # Switch to latest LTS
cue version install python 3.12  # Install new version
cue version remove python 3.9   # Remove old version
cue version pin                  # Write .python-version / .nvmrc to cwd
cue version current             # Show active versions of all managed runtimes
```

**Backend mapping:**

| Runtime | Version Manager |
|---------|----------------|
| Python | pyenv |
| Node.js | nvm / nvm-windows |
| Ruby | rbenv |
| Java | SDKMAN! |
| Go | gvm or binary swap |
| Rust | rustup |
| Terraform | tfenv |

`cue version` wraps these tools in a consistent interface — you never need to remember whether to use `pyenv global` or `nvm use` or `sdk use java`.

### 7.3 Environment Profiles (`cue profile`)

Profiles solve the multi-context problem for developers who maintain separate work and personal identities, or who work on fundamentally different project types requiring different PATH entries, git config, and SSH keys.

```bash
cue profile create work
cue profile switch work
cue profile list
cue profile diff work personal
cue profile current
```

**Profile Config (`~/.cue/profiles/work.toml`):**

```toml
[identity]
name = "work"

[git]
name = "Full Name"
email = "name@company.com"
signing_key = "~/.ssh/id_ed25519_work.pub"

[env]
PATH_prepend = ["/opt/company-tools/bin"]
AWS_PROFILE = "work"
DOCKER_HOST = "tcp://internal.docker.host:2376"

[shell]
default_tag = "work"
prompt_prefix = "[WORK]"
```

### 7.4 Team Sync (`cue team`)

Allows a team to maintain shared macros, stores, and config defaults distributed via a shared GitHub repository.

```bash
cue team init
cue team connect --repo <github-url>
cue team sync
cue team sync --push
cue team disconnect
```

**Team Repo Structure:**

```
team-devtools/
├── macros/
│   ├── deployment.toml
│   └── testing.toml
├── stores/
│   └── company-stack.toml
└── config/
    └── defaults.toml
```

Team settings take precedence over personal defaults but not over explicit user overrides.

### 7.5 Scheduled Maintenance (`cue schedule`)

```bash
cue schedule backup --interval daily
cue schedule doctor --interval weekly
cue schedule update-check --interval daily
cue schedule list
cue schedule remove <task-id>
```

Uses OS-native scheduling with no custom daemon: `systemd` user timers on Linux, `launchd` on macOS, Windows Task Scheduler on Windows.

## 8. Pillar 6 — Project Awareness Layer

### 8.1 `cue detect`

Auto-detects the project type in the current directory.

```
cd ~/projects/some-repo
cue detect

  Found: package.json + next.config.js → Next.js project
  Found: prisma/schema.prisma          → Prisma ORM (PostgreSQL)
  Found: docker-compose.yml            → Containerised services

  Project type: MERN / Full-stack JS

  Recommended store   : cue store install mern
  Missing tools       : pm2, pg_client

  Missing env vars (from .env.example vs .env):
  ✗ DATABASE_URL  (not set)
  ✗ JWT_SECRET    (not set)
  ✔ PORT          (set)
```

**Detection Signal Map:**

| File(s) Found | Detected Stack | Suggested Store |
|--------------|---------------|-----------------|
| `package.json` + `next.config.js` | Next.js | `frontend` |
| `package.json` + `mongoose` dep | MERN | `mern` |
| `requirements.txt` + `*.ipynb` | Data Science | `data-science` |
| `Cargo.toml` | Rust | `rust` |
| `go.mod` | Go | `golang` |
| `pom.xml` or `build.gradle` | Java/JVM | `java` |
| `*.tf` files | Terraform | `devops` |
| `hardhat.config.js` | Web3 | `web3` |
| `pubspec.yaml` | Flutter | `mobile` |

---

### 8.2 `cue init <project-name>`

Project scaffolding with stack-aware templates. Creates the directory, installs required tools, initialises git, and writes a `.cue` project config file.

```
cue init my-api

What kind of project?
  ● REST API (Node/Express)
  ○ REST API (Go/Gin)
  ○ REST API (Python/FastAPI)
  ○ Static site (Next.js)
  ○ Data Science notebook

→ Creating my-api/
→ Initialising git repository
→ Writing .gitignore
→ Writing .cue (tag: my-api, stack: mern)
→ Checking required tools... pm2 missing — installing
→ Scaffolding project structure

Project ready.
```

---

### 8.3 `.cue` Project File

A small TOML file in a project root that activates project-aware behaviour on `cd` (requires shell hook):

```toml
# .cue
tag = "my-api"
stack = "mern"
version_pins = { node = "20", python = "3.12" }
pre_run_checks = ["docker info", "mongod --version"]
```

On `cd` into a directory with this file: history tag switches, version pins activate, pre-run checks execute and warn on failure.

`cue model status` always shows proxy state (running / stopped / port conflict).

## 11. Out of Scope — v2.0 (v3.0 Candidates)

**GUI / Dashboard.** A browser-based dashboard for history, macros, and stores. The TUI is as far as the interactive layer goes in v2.0.

**AI-powered command suggestion.** When the user describes a goal in plain English and the tool suggests the right macro. Requires either an LLM call (breaks zero-dependency mandate) or a local sentence embedding model — possible as an opt-in in v3.0.

**Remote server management.** Extending `cue` to manage installations on remote machines via SSH. Significant scope and attack surface; better served by Ansible (already in the DevOps store).

**Container-first dev environments.** Integration with `devcontainers`, Nix flakes, or Devbox. Architecturally different from the current host-install model.

**WSL2 deep integration.** A `cue wsl` sub-command for configuring WSL2 itself. Currently WSL2 is treated as a standard Linux target.

**Encrypted workspace backup.** End-to-end encryption (age/GPG) before pushing to GitHub. Valid hardening; low urgency for most users.
